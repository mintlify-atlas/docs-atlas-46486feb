---
title: "Quickstart"
description: "Go from zero to a running agent in under five minutes."
---

<Steps>
  <Step title="Install gitagent">
    Install the CLI globally via npm. Node.js 18 or later is required.

    ```bash
    npm install -g gitagent
    ```

    Verify the installation:

    ```bash
    gitagent --version
    ```

    You should see the current version (e.g. `0.1.7`).
  </Step>

  <Step title="Scaffold a new agent">
    Run `init` with the `standard` template to create a fully structured agent directory in your current folder:

    ```bash
    gitagent init --template standard
    ```

    The command creates the following files:

    ```
    .
    ├── agent.yaml                      # Manifest — identity, model, skills, tools
    ├── SOUL.md                         # Identity, personality, values
    ├── RULES.md                        # Hard constraints and safety boundaries
    ├── AGENTS.md                       # Framework-agnostic fallback instructions
    ├── skills/
    │   └── example-skill/
    │       └── SKILL.md                # Starter skill with frontmatter
    ├── knowledge/
    │   └── index.yaml                  # Knowledge document index
    └── tools/                          # MCP-compatible tool definitions (empty)
    ```

    Only `agent.yaml` and `SOUL.md` are required by the spec. The rest is optional scaffolding you can fill in or delete.

    Open `agent.yaml` and set your agent's name, description, and model:

    ```yaml
    spec_version: "0.1.0"
    name: my-agent
    version: 0.1.0
    description: A helpful assistant agent
    author: ""
    license: MIT
    model:
      preferred: claude-sonnet-4-5-20250929
      constraints:
        temperature: 0.3
        max_tokens: 4096
    skills: []
    tools: []
    runtime:
      max_turns: 30
      timeout: 120
    tags: []
    ```

    Then open `SOUL.md` and define your agent's identity:

    ```markdown
    # Soul

    ## Core Identity
    I am a helpful assistant specializing in [your domain].

    ## Communication Style
    Clear, concise, and professional. I adapt my tone to the context.

    ## Values & Principles
    - Accuracy over speed
    - Transparency in reasoning
    - Helpfulness without overstepping
    ```
  </Step>

  <Step title="Validate your agent">
    Run `validate` to check your agent against the gitagent specification:

    ```bash
    gitagent validate
    ```

    For a freshly scaffolded agent, you'll see output like:

    ```
    Validating gitagent
    Directory: /path/to/your/agent
    ────────────────────────────────
    ✔ agent.yaml — valid
    ✔ SOUL.md — valid
    ✔ skills/ — valid
    ────────────────────────────────
    ✔ Validation passed (0 warnings)
    ```

    The validator checks:
    - `agent.yaml` schema correctness and that all referenced skills, tools, and sub-agents exist on disk
    - `SOUL.md` exists and contains real content (not just headings)
    - `skills/` — frontmatter validity, name matches directory, description length, instruction size
    - `hooks/hooks.yaml` and referenced scripts (if present)
    - `tools/*.yaml` schema correctness (if present)

    To also check regulatory compliance configuration:

    ```bash
    gitagent validate --compliance
    ```
  </Step>

  <Step title="Inspect your agent">
    `gitagent info` prints a formatted summary of everything gitagent knows about your agent:

    ```bash
    gitagent info
    ```

    The output includes:
    - Name, version, description, author, and license
    - Model preferences and constraints
    - Skills and tools listed in the manifest
    - Sub-agents defined in `agents/`
    - Runtime configuration (max turns, timeout)
    - Compliance settings (risk tier, frameworks)
    - Tags
    - A preview of `SOUL.md`

    To inspect an agent in a different directory:

    ```bash
    gitagent info -d ./examples/standard
    ```
  </Step>

  <Step title="Run your agent">
    Run your agent interactively using the Claude adapter (requires the [Claude Code CLI](https://docs.anthropic.com/en/docs/claude-code)):

    ```bash
    gitagent run -d . --adapter claude
    ```

    gitagent assembles a system prompt from your `SOUL.md`, `RULES.md`, skills, knowledge, and compliance settings, then passes everything to Claude Code via `--append-system-prompt`.

    To run with a one-shot prompt instead of interactive mode:

    ```bash
    gitagent run -d . --adapter claude -p "Review my authentication code"
    ```

    You can also run an agent directly from a git repository:

    ```bash
    gitagent run -r https://github.com/open-gitagent/gitagent --adapter claude
    ```

    <Note>
      `gitagent run` supports multiple adapters. Use `--adapter github` with a `GITHUB_TOKEN`, `--adapter openai` with an `OPENAI_API_KEY`, or `--adapter lyzr` with a `LYZR_API_KEY` to run against different backends without changing your agent definition.
    </Note>
  </Step>

  <Step title="Export your agent">
    Export your agent definition to a portable system prompt that works with any LLM:

    ```bash
    gitagent export --format system-prompt
    ```

    This concatenates (in order):
    1. Agent identity header (name + version)
    2. `SOUL.md` content
    3. `RULES.md` content
    4. Skills with instructions and allowed tools
    5. Knowledge documents flagged `always_load` in `knowledge/index.yaml`
    6. Compliance constraints as behavioral rules
    7. Memory content (if `memory/MEMORY.md` has content)

    Save the output to a file:

    ```bash
    gitagent export --format system-prompt --output system-prompt.md
    ```

    Other export formats let you target specific frameworks:

    ```bash
    # Claude Code CLAUDE.md
    gitagent export --format claude-code --output CLAUDE.md

    # OpenAI Agents SDK Python code
    gitagent export --format openai --output agent.py

    # CrewAI YAML configuration
    gitagent export --format crewai

    # GitHub Models API payload
    gitagent export --format github
    ```
  </Step>
</Steps>
