GitAgent is built on a single idea: **your repository is your agent**. Drop a handful of files into any git repo and it becomes a complete, portable agent definition that runs on any framework.

This section walks through every concept you need to understand that model — from the files that make up an agent, to skills, tools, memory, hooks, and composition.

## The "your repo is your agent" concept

Traditional AI agent frameworks lock your agent definition inside their own format. Moving from one framework to another means rebuilding from scratch.

GitAgent separates the _definition_ of an agent from its _execution_. An agent definition is just files in a directory:

```
my-agent/
├── agent.yaml    # who I am and how I should run
└── SOUL.md       # my identity and personality
```

That's it. Those two files are a valid, runnable agent. You version it with git, share it like any open-source project, and run it on Claude Code, OpenAI, CrewAI, Lyzr, GitHub Models, or any other framework using `gitagent run`.

Because the definition is just files, you get everything git gives you for free: commit history, branches, diffs, pull requests, tags, and forks.

## How agent files map to behavior

Every file in a GitAgent directory serves a specific role at runtime:

| File / Directory | What it controls |
|---|---|
| `agent.yaml` | Identity, model preferences, capability lists, runtime limits, compliance config |
| `SOUL.md` | Personality, tone, values, domain expertise — who the agent *is* |
| `RULES.md` | Hard constraints — what the agent must always and must never do |
| `DUTIES.md` | Segregation of duties policy and role boundaries |
| `skills/` | Reusable capability modules the agent can activate |
| `tools/` | MCP-compatible tool definitions the agent can call |
| `workflows/` | Multi-step procedures the agent can execute |
| `knowledge/` | Reference documents the agent can consult |
| `memory/` | Persistent state that survives across sessions |
| `hooks/` | Lifecycle event handlers (audit logging, compliance checks, escalation) |
| `agents/` | Sub-agent definitions for delegation |
| `compliance/` | Regulatory artifacts for audited environments |

When you run `gitagent export` or `gitagent run`, the CLI reads these files and assembles them into a system prompt or framework-specific configuration. Nothing is hard-coded — the files are the agent.
